# fromVillage PRD

## 1. 문서 목적

본 문서는 fromVillage MVP의 기능 요구사항과 비기능 요구사항을 정의한다.
본 프로젝트는 직거래 커머스 도메인을 기반으로 인증/인가/보안, 커머스 CRUD, 선착순 쿠폰 동시성 처리, 정산 배치를 핵심 기술 포인트로 삼는다.

## 2. 제품 목표

- 지역 생산자와 도시 소비자를 연결하는 직거래 플랫폼 MVP 구현
- Spring Security 기반 인증/인가 구조 설계
- 장바구니, 주문, 재고, 쿠폰을 포함한 커머스 핵심 흐름 구현
- Redis + DB 기반 선착순 쿠폰 발급 동시성 처리
- 관리자 수동 실행 기반 정산 배치 및 실행 이력 기능 구현

## 3. 사용자 유형

### USER

상품 탐색, 장바구니, 주문, 쿠폰 발급 및 사용, 본인 주문 취소를 수행한다.

### SELLER

본인 상품 등록, 수정, 삭제, 재고 관리, 본인 판매자 주문 조회를 수행한다.
MVP에서는 USER 구매 기능을 겸하지 않는 판매 전용 역할로 간주한다.
ADMIN 승인 시 USER에서 SELLER로 역할이 전환된다.

### ADMIN

판매자 권한 부여, 회원/상품/주문 조회, 쿠폰 정책 운영, 정산 배치 실행을 수행한다.
ADMIN은 운영 전용 계정으로 간주하며 일반 구매나 판매 행위는 수행하지 않는다.
ADMIN 계정은 공개 회원가입으로 생성하지 않고, 초기 시드 또는 운영 초기화 절차로만 생성한다.

## 4. MVP 기능 범위

### 4.1 회원 및 인증

- 회원가입
- 회원가입 비밀번호는 8자 이상 20자 이하이어야 한다.
- 회원가입 비밀번호는 영문 대문자, 영문 소문자, 숫자, 특수문자 중 3종 이상을 포함해야 한다.
- 회원가입 비밀번호에는 공백을 허용하지 않는다.
- 로그인
- 로그아웃
- 세션 기반 인증 상태 유지
- CSRF 토큰 조회
- 로그인 실패 보호 정책 적용

### 4.2 상품

- SELLER는 상품 등록, 수정, 삭제가 가능하다.
- SELLER는 본인 상품 목록을 조회할 수 있어야 한다.
- 상품 삭제는 물리 삭제가 아니라 soft delete로 처리해야 한다.
- 상품 카테고리는 농산물, 수산물만 지원한다.
- 상품 상태는 `판매중`, `품절`만 지원한다.
- 상품 가격은 정가만 지원한다.
- 재고 수량 관리가 가능해야 한다.
- 상품 대표 이미지 URL은 `https` 스킴만 허용해야 한다.
- 서버는 외부 `imageUrl`을 직접 fetch하지 않아야 한다.
- USER는 상품 목록 조회와 상세 조회가 가능해야 한다.
- soft delete된 상품은 공개 상품 조회, 장바구니 담기, 바로 구매, 체크아웃 대상에서 제외되어야 한다.
- soft delete된 상품의 공개 상세 조회는 `404 Not Found`로 응답해야 한다.

### 4.3 장바구니

- USER는 상품을 장바구니에 담을 수 있다.
- USER는 장바구니 상품 수량을 변경할 수 있다.
- USER는 장바구니 상품을 삭제할 수 있다.
- 장바구니에는 여러 SELLER의 상품을 함께 담을 수 있다.
- 장바구니 체크아웃이 성공하면 해당 `checkout_order`에 포함된 장바구니 항목만 삭제되어야 한다.

### 4.4 주문 및 결제

- USER는 장바구니 기반으로 체크아웃할 수 있다.
- USER는 바로 구매를 사용할 수 있다.
- 사용자 관점에서는 한 번의 체크아웃으로 하나의 주문이 생성된다.
- 내부적으로는 SELLER별 주문으로 분리된다.
- 바로 구매는 장바구니 항목을 생성하거나 삭제하지 않는다.
- 고객 관점 주문 상태는 `CREATED`, `COMPLETED`, `CANCELED`를 사용한다.
- 모의 결제 완료 시 주문은 즉시 `COMPLETED` 상태가 된다.
- 한 번의 체크아웃 주문에는 쿠폰 1장만 적용 가능하다.
- 쿠폰은 내부적으로 분리된 SELLER 주문 1건에만 적용 가능하다.
- 장바구니 체크아웃에서는 쿠폰 적용 대상 SELLER 주문을 요청의 `targetSellerId`로 명시해야 한다.
- 바로 구매에서는 `productId`로 SELLER가 결정되므로 `targetSellerId`를 별도로 받지 않는다.
- 쿠폰을 사용하지 않는 주문에서는 `issuedCouponId`와 `targetSellerId`를 전달하지 않는다.

### 4.5 주문 취소

- USER는 본인 주문만 취소 가능하다.
- SELLER와 ADMIN은 MVP에서 주문 취소 권한이 없다.
- 주문 취소는 `COMPLETED` 상태 주문에 대해서만 가능하다.
- `checkout_order` 취소 시 하위의 모든 `seller_order`도 함께 `CANCELED`로 전이되어야 한다.
- 주문 취소 시 재고는 복구되어야 한다.
- 쿠폰 사용 주문 취소 시 사용한 쿠폰도 복구되어야 한다.

### 4.6 쿠폰

- ADMIN은 플랫폼 공통 선착순 쿠폰 정책을 생성, 오픈, 종료할 수 있다.
- USER는 쿠폰 발급을 직접 요청할 수 있다.
- 쿠폰은 정액 할인만 지원한다.
- 쿠폰 정책은 최소 주문 금액, 유효기간, 발급 수량을 가진다.
- 쿠폰 최소 주문 금액 조건은 할인 전 `seller_order.total_amount`를 기준으로 검증해야 한다.
- 동일 USER는 동일 쿠폰을 중복 발급받을 수 없다.
- 쿠폰 수량 초과 발급은 허용되지 않는다.
- 발급된 개인 쿠폰은 동시에 하나의 SELLER 주문에만 연결될 수 있다.
- 쿠폰 발급 이력은 DB에 저장되어야 한다.

### 4.7 정산

- 정산은 ADMIN이 수동으로 실행한다.
- 정산 대상은 `COMPLETED` 상태이면서 아직 정산되지 않은 `seller_order`다.
- 정산 배치 실행 결과는 이력으로 저장되어야 한다.
- 정산은 SELLER 기준으로 계산되어야 한다.
- MVP에서 정산 금액은 `order_amount - discount_amount`로 계산하며, 플랫폼 수수료는 제외한다.

## 5. 주요 비즈니스 규칙

- 가입 기본 역할은 USER다.
- SELLER 권한은 ADMIN이 부여하며, 승인 시 USER에서 SELLER로 역할이 전환된다.
- USER가 이미 SELLER로 전환된 뒤에는 같은 계정을 다시 판매자로 승인하지 않는다.
- SELLER는 본인 상품과 관련된 정보만 관리할 수 있다.
- SELLER의 판매자 주문 조회는 주문 처리에 필요한 최소 정보만 제공한다.
- SELLER에게 노출되는 구매자 정보는 닉네임까지만 허용하며, 이메일/전화번호/주소 등 민감 정보는 제공하지 않는다.
- ADMIN은 회원 기본 정보와 역할 상태만 조회할 수 있으며, 민감 인증 정보는 조회할 수 없다.
- ADMIN은 운영 기능만 수행하며 장바구니, 주문 생성, 쿠폰 발급, 상품 등록 기능은 사용하지 않는다.
- 회원 계정은 상품과 달리 soft delete를 기본 규칙으로 두지 않는다. 탈퇴/비활성화 정책은 별도 계정 lifecycle 이슈에서 정의한다.
- 재고가 0이면 상품은 품절 상태가 된다.
- 재고 차감 결과가 0이 되면 같은 트랜잭션 안에서 상품 상태를 즉시 `SOLD_OUT`으로 전환해야 한다.
- 주문 취소로 재고가 다시 1 이상이 되면 같은 트랜잭션 안에서 상품 상태를 `ON_SALE`로 복구해야 한다.
- soft delete된 상품은 기존 주문 이력 보존을 위해 DB에 남아 있어야 한다.
- 주문 완료 시 재고가 차감된다.
- 주문 취소 시 재고가 복구된다.
- 고객 관점 주문 상세는 체크아웃 단위를 기준으로 조회한다.
- 쿠폰 사용 조건을 만족하지 못하면 주문에 적용할 수 없다.
- 유효기간이 지난 쿠폰은 사용할 수 없다.
- 이미 정산 결과가 생성된 주문은 중복 정산 대상이 될 수 없다.

## 6. 비기능 요구사항

- 인증이 필요한 API는 서버 세션 기반으로 보호되어야 한다.
- 세션은 Spring Session + Redis에 저장되어야 한다.
- 세션 식별자는 쿠키로만 전달되어야 하며, 응답 바디나 URL에 노출되어서는 안 된다.
- CSRF 보호는 활성화되어야 하며, 상태 변경 요청은 CSRF 토큰 검증을 거쳐야 한다.
- 로그인 성공 시 세션 ID가 재발급되어야 한다.
- 세션 쿠키 속성, 세션 만료 정책, 동시 세션 제한, 로그인 잠금 정책, 인증/인가 에러 코드 경계는 `docs/adr/001-auth-strategy.md`를 따른다.
- 회원가입/로그인 응답에는 내부 bigint 식별자(`userId`)를 노출하지 않아야 한다.
- 역할별 접근 제어가 적용되어야 한다.
- 본인 리소스 접근 제어가 가능해야 한다.
- 선착순 쿠폰 발급 시 동시 요청 환경에서도 정합성을 유지해야 한다.
- 정산 배치는 재실행 시 중복 정산을 방지해야 한다.
- 예외 응답은 일관된 형식으로 제공되어야 한다.

- 환불 및 반품

- 자동 스케줄 정산

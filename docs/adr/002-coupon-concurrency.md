# ADR-002 쿠폰 동시성 처리 전략

## 1. 상태

승인됨

## 2. 문맥

fromVillage의 핵심 기술 주제 중 하나는 선착순 쿠폰 발급 동시성 처리다.
여러 사용자가 동시에 쿠폰 발급을 요청할 때 중복 발급과 수량 초과 발급을 방지해야 한다.
동시에 최종 발급 이력은 영속 데이터로 안전하게 남겨야 한다.

## 3. 결정

쿠폰 발급은 Redis + DB 혼합 구조를 채택한다.

- Redis는 선착순 수량 제어와 중복 발급 선검증에 사용한다.
- DB는 최종 발급 이력 저장의 기준 데이터 저장소로 사용한다.
- DB 저장 실패 시 Redis 상태를 보정하는 흐름을 문서와 구현에 포함한다.
- 발급 수량의 최종 정합성 기준은 DB의 `issued_quantity`이며, Redis 카운터는 불일치 시 DB 기준으로 재조정한다.

## 4. 근거

Spring Data Redis는 `RedisTemplate`과 `SessionCallback`을 통해 동일 연결에서 MULTI/EXEC 기반 트랜잭션을 수행할 수 있고, 원자적 연산과 빠른 응답 특성을 제공한다.
이 특성은 선착순 수량 차감과 중복 발급 제어처럼 짧고 경쟁이 심한 연산에 적합하다.

반면 최종 발급 이력과 사용자별 쿠폰 상태는 관계형 데이터로 남겨야 조회, 복구, 주문 연동이 안정적이다.
따라서 Redis를 전면 제어 계층으로, DB를 최종 기준 저장소로 두는 구조가 가장 균형이 좋다.

## 5. 대안

### DB 잠금 기반 단일 처리

장점:
- 구조가 단순하다.
- 저장소가 하나라서 이해하기 쉽다.

단점:
- 동시 요청이 몰릴 때 경쟁 비용이 커질 수 있다.
- 포트폴리오의 동시성 포인트가 약해질 수 있다.

### Redis 단독 처리

장점:
- 빠른 처리 성능을 기대할 수 있다.

단점:
- 최종 발급 이력과 주문 연동을 영속적으로 설명하기 어렵다.
- 장애 복구 및 조회 요구에 약하다.

## 6. 결과

- 수량 초과 발급과 중복 발급을 방지할 수 있다.
- Redis/DB 간 보정 로직이 구현 난이도의 핵심이 된다.
- 동시성 테스트와 실패 보정 테스트가 중요해진다.

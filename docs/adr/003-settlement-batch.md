# ADR-003 정산 배치 처리 전략

## 1. 상태

승인됨

## 2. 문맥

fromVillage는 판매자별 정산을 핵심 기술 주제로 포함한다.
정산은 주문 완료 데이터 중 아직 정산되지 않은 항목만 대상으로 해야 하며, 재실행 시 중복 정산을 방지해야 한다.
또한 실행 이력과 결과를 분리해 운영 관점의 추적 가능성을 확보해야 한다.

## 3. 결정

정산 기능은 Spring Batch 기반으로 구현한다.

- 정산 실행 단위는 `Job`으로 표현한다.
- 대상 조회 및 처리 단위는 `Step`으로 표현한다.
- 주문 처리와 결과 저장은 chunk 기반으로 수행한다.
- 재실행 시 이미 정산된 주문은 제외한다.
- 실행 이력은 `settlement_batch`, 결과는 `settlement`에 분리 저장한다.
- MVP에서는 실제 지급 완료 워크플로우까지 다루지 않고, 정산 결과 생성까지만 범위에 포함한다.

## 4. 근거

Spring Batch는 Job/Step/Chunk 구조를 기본으로 제공하고, restartability, retry, skip 같은 배치 처리 핵심 기능을 지원한다.
정산처럼 대상 선정, 반복 처리, 결과 저장, 실패 이력 관리가 중요한 작업에는 일반 서비스 호출보다 배치 프레임워크가 더 명확한 구조를 제공한다.

이 프로젝트는 포트폴리오에서 배치 설계와 운영성까지 보여주는 것이 목적이므로, Spring Batch를 명시적으로 채택하는 것이 적절하다.

## 5. 대안

### 일반 서비스 메서드 기반 수동 실행

장점:
- 초기 구현이 단순하다.

단점:
- 배치 구조와 재실행 정책을 설명하기 어렵다.
- 이력 관리와 실패 처리 전략이 애매해질 수 있다.

### 단순 스케줄러 기반 처리

장점:
- 실행 트리거는 단순하다.

단점:
- 이번 MVP는 수동 실행 + 실행 이력 조회가 요구사항이라 맞지 않는다.
- 배치 처리 구조를 드러내기 어렵다.

## 6. 결과

- 정산 대상 선정과 결과 저장 흐름을 명확히 문서화할 수 있다.
- 재실행 안정성과 중복 방지 규칙을 구조적으로 설명할 수 있다.
- 지급 완료 단계는 후속 확장 과제로 남겨두고, 현재 MVP 범위를 안정적으로 유지할 수 있다.
- 배치 테스트가 핵심 품질 게이트가 된다.

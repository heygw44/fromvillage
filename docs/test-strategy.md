# fromVillage Test Strategy

## 1. 문서 목적

본 문서는 fromVillage MVP의 테스트 전략을 정의한다.
프로젝트 핵심 주제인 인증/인가/보안, 주문 정합성, 쿠폰 동시성, 정산 배치를 검증 가능한 형태로 분해한다.

## 2. 테스트 원칙

- 기능 구현 전에 실패 테스트를 먼저 정의한다.
- 도메인 규칙은 단위 테스트로 빠르게 검증한다.
- API 계약과 권한 제어는 통합 테스트로 검증한다.
- 쿠폰 발급은 동시성 테스트로 정합성을 증명한다.
- 정산 배치는 배치 테스트로 재실행 안정성과 중복 방지를 확인한다.
- 성공 케이스와 실패 케이스를 항상 함께 작성한다.

## 3. 테스트 4축

### 3.1 단위 테스트

목표:
- 도메인 규칙과 상태 전이를 빠르게 검증한다.

주요 대상:
- 세션 사용자 식별 및 권한 판별 보조 로직
- `Order` 상태 전이 (`CREATED -> COMPLETED -> CANCELED`)
- `checkout_order` 취소 시 하위 `seller_order` cascade 취소 규칙
- 주문 취소 시 재고 복구 규칙
- 재고가 0이 될 때 `SOLD_OUT`으로, 재고 복구 후 1 이상이 될 때 `ON_SALE`로 자동 전환되는지 검증
- 쿠폰 사용 가능 조건 (유효기간, 최소 주문 금액, 상태)
- 쿠폰 복구 규칙 (`USED -> ISSUED`)
- 정산 금액 계산 로직 (`order_amount - discount_amount`)

성격:
- 외부 I/O 없이 도메인 객체/정책 단위 검증
- 빠른 실행, 높은 회귀 탐지율

### 3.2 통합 테스트

목표:
- API 계약, 권한, 트랜잭션 경계를 검증한다.

주요 대상:
- 인증 API (`signup`, `login`, `logout`, `csrf`)
- 회원가입 비밀번호 길이/조합/공백 검증 규칙
  예: 7자 실패, 8자 성공, 공백 포함 실패, 2종 조합 실패
- URL 보안 + 메서드 보안 동시 검증
- USER/SELLER/ADMIN 권한별 접근 제어
- SELLER 본인 상품 목록 조회 API 권한 및 소유권 검증
- 세션 생성 및 로그인 성공 후 세션 ID 재발급 검증
- `docs/adr/001-auth-strategy.md`에 정의된 세션 쿠키 속성, 세션 만료, 동시 세션 제한, 로그인 잠금 정책 검증
- Redis에 저장된 실패 횟수와 잠금 만료 시각 갱신 검증
- 로그인 잠금 상태 요청의 `401` 및 `AUTH_LOGIN_TEMPORARILY_LOCKED` 응답 검증
- 상태 변경 요청의 CSRF 검증과 로그인/로그아웃 후 CSRF 재조회 흐름 검증
- 미인증 요청의 `401 + AUTH_UNAUTHORIZED`, 세션 만료 요청의 `401 + AUTH_SESSION_EXPIRED`, CSRF 실패의 `403 + AUTH_CSRF_INVALID`, 권한 부족 요청의 `403 + AUTH_FORBIDDEN` 응답 검증
- 세션 쿠키가 없는 요청과 만료된 세션 쿠키 요청을 분리해 각각 `AUTH_UNAUTHORIZED`, `AUTH_SESSION_EXPIRED`로 응답하는지 검증
- 장바구니 체크아웃/바로구매 공통 유스케이스 검증
- 주문 생성 시 판매자별 주문 분리 검증
- 장바구니 체크아웃 성공 시 포함된 `cart_item`만 삭제되고, 바로 구매는 장바구니를 변경하지 않는지 검증
- 장바구니 체크아웃은 `targetSellerId`를 사용하고, 바로 구매는 `productId`만으로 SELLER를 결정하는지 검증
- 쿠폰 미사용 주문에서 `issuedCouponId`와 `targetSellerId` 생략이 허용되는지 검증
- 쿠폰 최소 주문 금액을 할인 전 `seller_order.total_amount` 기준으로 검증하는지 확인
- 상품 `imageUrl`의 `https` 검증 및 서버 비-fetch 정책 검증
- soft delete된 상품이 공개 조회와 신규 주문 흐름에서 제외되는지 검증
- soft delete된 상품 상세 조회가 `404 Not Found`로 응답하는지 검증
- 공통 응답 래퍼와 에러 계약 검증

성격:
- Controller -> Service -> Repository 흐름 포함
- 실제 DB를 사용하거나 테스트 컨테이너 기반으로 검증

### 3.3 동시성 테스트

목표:
- 선착순 쿠폰 발급의 정합성을 증명한다.

주요 대상:
- 동일 쿠폰에 대한 대량 동시 발급 요청
- 중복 발급 방지 (`coupon_policy_id`, `user_id` 유일성)
- 발급 수량 상한 초과 방지
- Redis 선검증 + DB 영속화 실패 시 보정 시나리오

성공 기준:
- 총 발급 건수는 정책 수량을 초과하지 않는다.
- 동일 사용자는 동일 쿠폰을 중복 발급받지 않는다.
- 실패 발생 후에도 Redis/DB 상태가 최종적으로 일관된다.

### 3.4 배치 테스트

목표:
- 정산 배치의 대상 선정, 중복 방지, 재실행 안정성을 검증한다.

주요 대상:
- 대상 선정: `COMPLETED` + 미정산 주문만 포함
- 배치 실행 이력(`settlement_batch`) 기록
- 판매자별 정산 결과(`settlement`) 생성
- 재실행 시 이미 정산된 주문 제외
- 실패 건 처리 및 실패 이력 기록

성격:
- Spring Batch Job/Step 단위 검증
- 데이터 준비 -> 배치 실행 -> 결과 검증 패턴 사용

## 4. 기능별 테스트 매핑

### 인증/인가/보안

- 단위: 세션 사용자 식별 유틸, 권한 판별 로직
- 통합: 보호 경로 접근 제어, 역할별 인가, 본인 리소스 접근 제어

### 상품/장바구니/주문

- 단위: 주문 금액 계산, 재고 차감/복구 규칙
- 통합: 체크아웃 API, 바로구매 API, 판매자별 주문 분리

### 쿠폰

- 단위: 사용 가능성 판단, 복구 상태 전이
- 통합: 발급/조회/적용 API 계약
- 동시성: 대량 요청 시 수량 및 중복 보장

### 정산

- 단위: 정산 금액 계산
- 통합: 관리자 배치 실행 API 권한 및 응답
- 배치: 대상 선정, 결과 생성, 재실행 안전성

## 5. 테스트 데이터 전략

- 역할별 기본 사용자 픽스처를 유지한다 (`USER`, `SELLER`, `ADMIN`).
- 주문/쿠폰/정산 데이터는 시나리오별 빌더로 생성한다.
- 시간 의존 규칙(유효기간, 정산 시점)은 테스트 전용 고정 시간으로 제어한다.
- 동시성 테스트는 독립된 쿠폰 정책과 사용자 집합을 사용한다.

## 6. 테스트 우선순위

1. 인증/인가 기본 통합 테스트
2. 주문 생성/취소 정합성 테스트
3. 쿠폰 동시성 테스트
4. 정산 배치 테스트
5. 세부 조회/검색 API 테스트

## 7. 품질 게이트

- 핵심 유스케이스(로그인, 주문, 쿠폰 발급, 정산 배치)는 회귀 테스트가 존재해야 한다.
- 동시성 테스트에서 수량 초과 또는 중복 발급이 1건이라도 발생하면 실패로 간주한다.
- 배치 재실행 테스트에서 중복 정산이 발생하면 실패로 간주한다.
- 공통 응답 형식 위반 API는 실패로 간주한다.
